* Search optimization analyzer
The =artifact_search_optimization_analysis= entry point materializes:
- =events= :: classified =fzf=, =rg=, =prime=, structural query, dependency, owner, and direct-source-read events
- =summary= :: counts and latency by search tool and query type
- =graph= :: typed graph over events, tools, query types, route hints, languages, providers, targets, and normalized query keys
- =metrics= :: PageRank and degree centrality for route-promotion graph nodes
- =opportunities= :: repeat-search, latency-hotspot, =fzf= route promotion, =rg= route promotion, generic-route classification gaps, and timestamp coverage gaps

This analyzer is the source-level behavior behind the modular Pluto report =notebooks/search_optimization_algorithms.jl=.

The fzf and rg research views are intentionally split:

- =notebooks/fzf_algorithm_research.jl= uses =artifact_fzf_algorithm_analysis= to study candidate-set pressure, fuzzy selection reuse, route-promotion opportunities, and missing candidate-count/rank instrumentation.
- =notebooks/rg_algorithm_research.jl= uses =artifact_rg_algorithm_analysis= to study pattern selectivity, path/language filtering, repeated scan pressure, and structural-query replacement.

The combined =search_optimization_algorithms.jl= notebook should remain an overview. Tool-specific algorithm work belongs in the split notebooks so fzf and rg can evolve with different hypotheses, measurements, and optimization gates.

* Improvement questions
- Which =fzf= paths should be promoted to typed owner/dependency routes?
- Which repeated =prime= calls are suppressible because owner topology already exists?
- Which repeated =rg= patterns should become indexed symbol/call/docs-use queries?
- Which =query/owner-items= calls are slow and concentrated on a small set of owners?
- Which direct-source reads indicate selector identity gaps?
- Which language providers need cache or harness health work before algorithm changes are meaningful?
